A comprehensive analysis of Gaussian Splatting Rasterization


Posted by Hwan Heo on August 08, 2024

Figure credit: Luma AI

Introduction

The biggest advantage of 3D Gaussian Splatting is its rendering 'speed' of over 100 fps. This speed, which is faster than any other NeRF-based method, is due to a well-designed tile-based rasterization. Let's try to understand 3D Gaussian Splatting from this rasterization perspective (as fully as possible).

1. 3D Gaussian as Primitive Kernel

The basic idea of 3D Gaussian Splatting is to represent a scene with explicit, learnable primitives, unlike implicit NeRF. (There are several advantages of using an explicit representation, which were mentioned in the previous 2D GS review. If you're curious, please refer to this link.)

The authors defined the kernel for these primitives (particles) with the following 3D Gaussian function.

$$G(x) = \exp \left( {- \frac{1}{2} x^{\rm T} \Sigma^{-1} x} \right )$$

You may ask, did this kernel design have to be 3D Gaussian? There are several reasons for this choice mentioned in the paper.

  • To be differentiable for training.

  • The projection to 2D is easy and well-defined for fast alpha blending.

If these criteria are met, it seems that any kernel design can be used, and in fact, recent experiment with different kernels for 3D Gaussian Ray Tracing found no significant difference in performance.

Since the covariance matrix only has physical meaning if it is positive definite, the authors suggest organizing the covariance matrix in the following form for ease of learning:

$$ \Sigma = RSS^{\rm T}R^{\rm T} $$

where $R,\ S \ \in \mathbb{R}^{3 \times 3}$ rotation, scale matrix.

We can rewrite this covariance matrix with a slight change as follows:

$$ \begin{aligned} \Sigma & = R (SS^{\rm T}) R^{\rm T} \\ & = R \begin{bmatrix} s_1^2 & 0 & 0 \\ 0 & s_2^2 & 0 \\ 0 & 0 & s_3^2 \\ \end{bmatrix} R^{\rm T} \end{aligned} $$

It can be seen that the covariance matrix has the same form as the 3D ellipsoid (anistropic) matrix, i.e., 3D GS uses an opaque ellipsoid in 3D (whose density is defined by a Gaussian distribution) as a primitive kernel.

Tip. When dealing with quadratic forms ($A^{-1}MA$), it is often useful to interpret $M$ as a transformation in $A$ coordinate system. This is also true for eigendecomposition, which is the analysis of how much weight each axis has (eigenvalues) in a coordinate system of axes (eigenvectors) whose orientation is preserved under linear transformation. Thinking about it this way, it's clear why PCA is related to eigendecomposition.

In other words, the defined Covariance matrix can be interpreted as:

  • An intensity matrix (squared scale) in the coordinate system on which each principal axis of the ellipsoid is basis

  • A method of expressing it in the world coordinate system 


                            
// compute 3D covariance matrix
glm::mat3 S = glm::mat3(1.0f);
S[0][0] = mod * scale.x;
S[1][1] = mod * scale.y;
S[2][2] = mod * scale.z;

glm::mat3 R = glm::mat3(
    1.f - 2.f * (y * y + z * z), 2.f * (x * y - r * z), 2.f * (x * z + r * y),
    2.f * (x * y + r * z), 1.f - 2.f * (x * x + z * z), 2.f * (y * z - r * x),
    2.f * (x * z - r * y), 2.f * (y * z + r * x), 1.f - 2.f * (x * x + y * y)
);

glm::mat3 M = S * R;
glm::mat3 Sigma = glm::transpose(M) * M;

cov3D[0] = Sigma[0][0];
cov3D[1] = Sigma[0][1];
cov3D[2] = Sigma[0][2];
cov3D[3] = Sigma[1][1];
cov3D[4] = Sigma[1][2];
cov3D[5] = Sigma[2][2];

Here, Note that

  • $S$: 3x3 diagonal matrix,
  • $R$: Calculated using the quaternion to rotation matrix formula
    $$ R = \begin{bmatrix} 1 - 2y^2 - 2z^2 & 2xy - 2zw & 2xz + 2yw \\ 2xy + 2zw & 1 - 2x^2 - 2z^2 & 2yz - 2xw \\ 2xz - 2yw & 2yz + 2xw & 1 - 2x^2 - 2y^2 \end{bmatrix} $$
  • COV: Since it is symmetric, we only need to store the right upper triangle.

2. Splatting (Projection of Primitives)

2.1. Projection

Now we need a way to project the covariance matrix of the Gaussian defined over the world system into image space, which the authors solve with the following method, proposed in EWA Splatting.

$$ \Sigma^{\prime} = JW\Sigma (JW)^{\rm T} $$

Let's analyze the meaning of this expression in a little more detail.

  • For this transformation, the same interpretation of quadratic forms applies: covariance matrix in world coordinate system → camera space → ray space.

  • Here, $J$ is the Jacobian (affine approximation) of the perspective projection (camera space to ray space), which means the $\phi$ transformation from camera space to ray space, represented as

    $$ \phi(x) = \phi(t) + J \cdot (x - t) $$

  • Since it is a 1st order Taylor approximation, there will be an approximation error as we move away from the center of the Gaussian. Recently, there are also studies that present this perspective error as a limitation of 3D GS and try to solve it. (2D Gaussian Splatting / On the error analysis of 3D Gaussian Splatting)

The Jacobian $J$ for the perspective projection is derived as follows:

\begin{bmatrix} 1 / t_2 & 0 & -t_0 / t_2^2 \\ 0 & 1 / t_2 & -t_1 / t_2^2 \\ t_0 / \| t\| & t_1 / \| t\| & t_2 / \| t\| \\ \end{bmatrix}

In a practical implementation, for considering following reason,

  • image space pixel $(u,v)$ in the pinhole camera model,

  • the required covariance matrix in image space is of 2x2, the 3rd row (z-axis) of the projected covariance is not actually used,

so, it is implemented in the form of

\begin{bmatrix} f_x / t_2 & 0 & -f_x \cdot t_0 / t_2^2 \\ 0 & f_y / t_2 & - f_y \cdot t_1 / t_2^2 \\ 0 & 0 & 0 \\ \end{bmatrix} 

                            
// affine approximation of the Jacobian matrix of viewmatrix to rayspace
glm::mat3 J = glm::mat3(
    focal_x / t.z, 0.0f, -(focal_x * t.x) / (t.z * t.z),
    0.0f, focal_y / t.z, -(focal_y * t.y) / (t.z * t.z),
    0, 0, 0);

// W: w2c matrix 
glm::mat3 W = glm::mat3(
    viewmatrix[0], viewmatrix[4], viewmatrix[8],
    viewmatrix[1], viewmatrix[5], viewmatrix[9],
    viewmatrix[2], viewmatrix[6], viewmatrix[10]);
    
glm::mat3 T = W * J;
glm::mat3 Vrk = glm::mat3(
    cov3D[0], cov3D[1], cov3D[2],
    cov3D[1], cov3D[3], cov3D[4],
    cov3D[2], cov3D[4], cov3D[5]);

glm::mat3 cov = glm::transpose(T) * glm::transpose(Vrk) * T;

2.2. Density of the Projected Gaussian

Since actual rendering uses a Gaussian density value multiplied by an opacity value, for a point $p$ in 3D space, the density $f_i(p)$ of the $i$th Gaussian can be defined as follows:

$$ f_i(p) = \sigma(\alpha_i) \exp \left( - \frac{1}{2} (\mu_i -p)^{\rm T} \Sigma_i^{-1} (\mu_i - p ) \right ) $$

  • This can also be viewed as a weighted (opacity) probability density function for a multivariate (3D) normal distribution.

  • The value of the inner exponential is the Mahalanobis Distance, which can be viewed as the distance (similarity) within an ellipsoid given the scale. This is exactly the intuition that for any 3D point, the closer to the Gaussian, the more opaque the Gaussian, the larger the response should be.

To calculate the density value above, we need the inverse matrix of the projected covariance. Let's take a look in the implementation perspective, which have a few linear algebra tricks to get the inverse.

Recap the earlier computation to get 2D covariance matrix. The actual end of this code is implemented as follows, 

// compute cov 2D
cov[0][0] += 0.3f;
cov[1][1] += 0.3f;

return { float(cov[0][0]), float(cov[0][1]), float(cov[1][1]) };

which is the same as finding the inverse of the following inverse matrix, $(A=(RS)^{\rm T})$

$$ A^{\rm T}A+\lambda \mathbf{I} $$

Since the covariance matrix is positive semidefinite, adding a small $\lambda$ makes the covariance matrix positive definite. $$ x^T A^T Ax + \lambda x^T x > 0 $$ This trick avoids numerical instability when finding the inverse matrix. 

// compute inverse of the covariance 2D
float det = (cov.x * cov.z - cov.y * cov.y);
if (det == 0.0f)
	return;
float det_inv = 1.f / det;
float3 conic = { cov.z * det_inv, -cov.y * det_inv, cov.x * det_inv };

  • Note that the inverse matirx formula for 2x2 matrix is $$ A=\begin{bmatrix} a & b \\ b & c \end{bmatrix}, \quad A^{-1} = \frac{1}{\det(A)} \begin{bmatrix} c & -b \\ -b & a \end{bmatrix} $$

  • Also note that the inverse of cov is called conic, presumably because the definition of conic section.


Finally, we calculate the radius of each splat, which can be covered at least 99.7% of the time $$ r = 3 \times \max_i \textit{standard deviation}_i $$ and use this value for Gaussian culling (masking) purposes. (Otherwise, you would have to query all Gaussians in the scene....)

The standard deviation of the 3D Gaussian defined by the Covariance matrix is equal to the eigenvalue, so we can use the following Characteristic equation to get the standard deviation. $$ \det \left ( A -\lambda \mathbf{I} \right ) = 0, \\ \rightarrow (a- \lambda)(c-\lambda) - b^2 = 0. $$

Since this is a quadratic equation for $\lambda$, it has the famous closed form solution :) The code also uses the root formula to find the value of the lambda. 

// compute inverse of the covariance 2D
float mid = 0.5f * (cov.x + cov.z);
float lambda1 = mid + sqrt(max(0.1f, mid * mid - det));
float lambda2 = mid - sqrt(max(0.1f, mid * mid - det));

float my_radius = ceil(3.f * sqrt(max(lambda1, lambda2)));

3. Rasterization

We know what a Gaussian kernel is, and how it is projected onto image space. Therefore,

  • for the Gaussians that intersect the ray.

  • collect the density and opacity values sorted by depth order only,

we can draw a 3D GS scene as a 2D image.

To make this rendering task efficient, 3D Gaussian Splatting proposed a tile-based rasterization that projects all 3D Gaussians onto an image space (screen), divides them into small tiles, and performs color/opacity accumulation for each tile in parallel.

Let's analyze the rasterization algorithm presented in the paper step-by-step.



  1. Divide the screen into 16x16 tiles.

    <<< total number of tiles, 256 >>> for CUDA parallelization and rasterization.

  2. Frustum culling to keep only valid Gaussians. (See fig.)


  3. Duplicate the Gaussians that overlap for each tile. (Instantiate)

  4. Sort the Gaussians (in each tile) in depth order (using GPU radix sort).
    • Since the covariance we use for rendering is already projected into image space, if we don't sort by depth, it will look like a bunch of overlapping Gaussians in no particular order.
    • This sorting is done before each thread block execution, so we don't sort on a per-tile basis, i.e. pre-sort primitives!

  5. Set the working range for each tile using the sorted Gaussian, and run one CUDA thread block per tile to rasterize in parallel.
    • Each thread block stores some information in shared memory to reduce memory bottlenecks.
    • The final color value is calculated by accumulating opacity and color along a sorted Gaussian.

The __shared__ variable is actually implemented to store the values of ID, opacity, pixel coordinates, etc. In other words, the Gaussian's projected covariance, spherical harmonics color, etc. are already calculated and stored in shared memory before the tile-by-tile calculation takes place.

// Allocate storage for batches of collectively fetched data.
__shared__ int collected_id[BLOCK_SIZE];
__shared__ float2 collected_xy[BLOCK_SIZE];
__shared__ float4 collected_conic_opacity[BLOCK_SIZE];

Also, the accumulation process for calculating the color in (5) can be written as a formula $$ C(\mathbf{x}) = \sum_{i \in N} T_i \, g_i^{2D}(x) \, \alpha_i \, \mathbf{c}_i , \quad \text{where } T_i = \prod_{j=1}^{i-1} \left(1 - g_j^{2D}(x) \, \alpha_j\right) $$ we can see the difference from NeRF.

  • Unlike NeRF, which requires sampling points along the ray and querying it to the MLP for getting the opacity/color of the point,

  • 3D GS sums the opacity/color while traversing $N$ splats already projected on the tile in depth order.

In other words, since NeRF samples rays, it queries the MLP for each different point sampled by $r(t)$ (3D), whereas 3D GS queries the MLP for the same point $x$ (2D).

5. Conclusion

In the case of NeRF, which leaves a large part of the algorithm to MLP, it is easy to analyze how the code is structured from the perspective of Linear Algebra or Computer Graphics. However, in the case of 3D Gaussian Splatting, since it deals with explicit primitives, the rasterization module is designed very delicately, so it seems to be quite difficult to analyze from a mathematical and code perspective.

This article focuses on the forward function of the diff-gaussian-rasterizer by the INRIA group, but the actual implementation also includes the calculation of RGB colors from Spherical Harmonics, allocation of CUDA thread blocks, and backward through the rasterization process. The backward is almost the inverse of the forward computation described above, computing the gradient while traversing back-to-front as opposed to front-to-back in the forward step.
